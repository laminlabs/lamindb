**Title:** Signal handler overrides `SIGINT` and prevents KeyboardInterrupt

**Body:**
While working with the `lamindb` library, I noticed an issue where the custom signal handler in the `LogStreamTracker` class overrides signals like `SIGINT` without properly re-raising the original exception (e.g., `KeyboardInterrupt`). As a result, when trying to interrupt a process manually with `Ctrl+C`, the process fails to terminate as expected because the exception is not propagated.

### Problematic Code:
```python
if threading.current_thread() == threading.main_thread():
    signal.signal(signal.SIGTERM, self.cleanup)
    signal.signal(signal.SIGINT, self.cleanup)
```
In the `cleanup` handler, relevant exceptions should typically be re-raised after the custom signal handling logic is executed. However:
```python
def cleanup(self, signo=None, frame=None):
    try:
        # Cleanup logic
        pass
    except:  # noqa: E722, S110
        pass
```
This snippet swallows any exception, causing unexpected behavior like blocking `KeyboardInterrupt`.

### Cause:
The `cleanup` method does not re-raise the `KeyboardInterrupt`. Furthermore, exceptions within the `cleanup` method are silenced by broad `try...except` statements.

### Steps to Reproduce:
1. Run a process managed by `lamindb`.
2. Attempt to terminate it using `Ctrl+C`.
3. Observe that the process does not terminate gracefully due to the missing re-raise.

### Suggested Fix:
To fix this, the `cleanup` method should re-raise the `KeyboardInterrupt` after its logic is executed. This ensures proper termination and debugging capability.

Example fix:
```python
def cleanup(self, signo=None, frame=None):
    try:
        # Cleanup logic
    finally:
        if signo == signal.SIGINT:
            raise KeyboardInterrupt
```
This small change ensures that interruptions like `Ctrl+C` are handled correctly.
